## One-time external setup

| What | Cost | Where | Notes |
|---|---|---|---|
| Apple Developer Program | $99/year | <https://developer.apple.com/programs/> | Verification 24–48h |
| Google Play Console | $25 one-time | <https://play.google.com/console/> | Verification a few hours to days |
| Expo / EAS | Free for our usage | <https://expo.dev/> | Already done — `faizanraza09` |

After Apple approval, grab the **Team ID** (top of <https://developer.apple.com/account>)
and fill it in at `eas.json` → `submit.production.ios.appleTeamId`. Also create
the App Store Connect app record (Bundle ID `pk.konsacard.app`) and copy the
**App Store Connect App ID** into `ascAppId`.

After Google Play setup, create a service account with **Service Account User**
+ **Release Manager** roles in Play Console, download the JSON key, save it as
`apps/mobile/play-service-account.json` (already gitignored).

## Build & test (preview profile)

These commands work right now without any store account.

```bash
cd apps/mobile

# APK for sideloading on Android
npm run build:preview -- --platform android

# Signed iOS build for TestFlight (needs Apple Developer)
npm run build:preview -- --platform ios
```

Each build takes ~15–20 min on EAS's free tier. You'll get an email and a
download link. Watch live at the URL printed in your terminal.

Install the APK by opening the download link on the device. Allow
"Install unknown apps" for the browser the first time.

iOS preview builds appear in TestFlight automatically; add testers in
App Store Connect → Users and Access → TestFlight (up to 100 internal,
no Apple review needed).

## Production builds & store submission

When you're satisfied with preview:

```bash
# Build store-ready binaries (~20 min each)
npm run build:prod                # both platforms in parallel
# or one at a time:
npm run build:prod:ios
npm run build:prod:android

# Upload to App Store Connect / Play Console
npm run submit:ios                # uploads to App Store Connect → review
npm run submit:android            # uploads to Play Console → internal track
```

Submit-step prerequisites: `eas.json` placeholders filled in (iOS) and
`play-service-account.json` present (Android).

Apple review: 24–72h typical; up to a week. Common rejection reasons:
- App requires a sign-in but doesn't provide demo credentials (we don't, so n/a)
- Broken deep link from privacy policy → must work from inside the app
- Screenshots use marketing copy that isn't in the app itself

Google Play first review: a few days. Updates usually under 24h after that.

## Store listing prep (do once, edit later in the console)

You'll need this content for both stores. Drafts can live in
`docs/store-listing.md` if you want to version-control them.

- **App name** (30 char): "KonsaCard" or "KonsaCard — Restaurant Deals"
- **Subtitle / short description** (30 char iOS, 80 char Android): e.g.
  "Best card per restaurant in PK"
- **Long description** (4000 char). Pitch + feature bullets + how it works.
  Don't claim things the app doesn't do.
- **Keywords** (iOS only, 100 char comma-separated): "credit card,debit card,
  discount,karachi,lahore,islamabad,restaurant,pakistan,banking"
- **Screenshots**:
  - iOS: 5–10 per device size, 6.7" iPhone is minimum. From an iOS simulator.
  - Android: 2–8, 16:9 portrait, plus a feature graphic (1024×500).
- **App icon**: 1024×1024 PNG (we have one at `assets/icon.png`)
- **Category**: Finance or Food & Drink — pick one and stick with it
- **Privacy policy URL**: <https://konsacard.pk/privacy-policy/>
- **Support URL**: <https://konsacard.pk/contact/>
- **Age rating questionnaire**: all "no" → 4+ / Everyone
- **App Tracking Transparency (iOS)**: answer "No, we do not track" — PostHog
  by itself doesn't require ATT since we don't link analytics to identity
- **Privacy nutrition labels (iOS) / Data safety (Android)**: most fields
  "Usage Data → Not linked to user". Matches what the privacy policy says.

## When to ship a new build vs. when not to

| Change | New mobile build? |
|---|---|
| Daily offers refresh | No — mobile fetches `konsacard.pk/data/*` at runtime |
| Card requirements update | No |
| Cuisine tagging changes | No |
| Mobile UI bug fix | Yes |
| New mobile feature | Yes |
| Algorithm change in `src/lib/algorithms.ts` | **Yes** — mobile has its own port; web's `lib/ranking-core.mjs` doesn't reach it |

The algorithm one is the sharp edge to remember. Web and mobile algorithms
must stay in sync manually.

## Gotchas seen so far

- **`npm ci` strict mode + peer conflicts:** the react / react-dom peer
  mismatch from `expo-router/vaul` forces us to use legacy resolution. The
  `.npmrc` here pins that for both local installs and EAS Build's `npm ci`.
  Don't delete `.npmrc`.
- **`expo doctor` blocks builds on non-zero exit:** if doctor flags any
  package version mismatch, EAS aborts before compilation. Fix with
  `npx expo install --fix` and re-queue.
- **First Android build prompts for keystore:** EAS auto-generated one. It's
  stored on Expo's side as `Build Credentials 2rF59NMZEj`. Future builds reuse
  it — critical for app upgrades since Play Store rejects key changes.
- **EAS Build uploads only `apps/mobile/`:** other parts of the repo
  (`apps/web`, scripts) aren't sent, so don't reference them from mobile code.
- **Expo Go warnings about push notifications:** harmless during dev — local
  notifications still work in Expo Go, only push token registration is
  removed (we don't need it to ship; can enable on dev-client builds later).

## Recurring release flow (after first launch)

1. Land code changes on `main` (Cloudflare auto-deploys web).
2. `cd apps/mobile && npm run build:prod` — kicks off both platforms.
3. ~20 min later: `npm run submit:ios && npm run submit:android`.
4. iOS: open App Store Connect, attach the new build to a new version, submit
   for review. Android: in Play Console, promote the build from the internal
   track to production.
5. Review wait: ≤24h Google, 24–72h Apple typical.
